# Chatbot 数据分析 Spec

## 背景

当前 chatbot（`public/app.js`）能查询和展示数据（商户详情、推荐排名、支付状态、品类列表），但无法回答分析类问题。当用户问"Shokz 最近表现怎么样？"或"美妆品类什么趋势？"时，系统只能退化为关键词搜索或直接找不到匹配。

项目已有 LLM 集成基础：`llm_classify.py` 通过 DeepSeek/Claude 做意图分类，`/api/chat/classify` 端点提供分类服务。但 LLM 仅用于意图分类（6 个标签+参数提取），不参与回答生成。

此外，`window.CHATBOT_DATA` 和 `window.SHEET_REPORT_DATA` 已包含丰富的商户指标、品类归属、Tier 等级、支付记录等结构化数据，具备分析的数据基础。

## 目标

为 chatbot 新增「数据分析师」能力，使其不仅能查数据，还能：

- 对商户、品类、Tier 三个维度进行描述性统计分析
- LLM 生成分析叙述文字和行动建议
- 用户可通过显式指令或自然提问触发分析
- 分析结果以「统计表格 + 文字解读」的混合形式呈现
- 复用前端内存中的现有数据，不引入新的后端 API 或图表库

## 功能需求

- **F1**: 新增「分析」意图类型——在现有 6 种意图（asin/merchant/payment/recommendation/tier/category）之外增加 `analysis` 意图。LLM 在用户提问涉及"分析、评估、诊断、趋势、对比、表现"等语义时返回 `analysis`。`llm_classify.py` 的 system prompt 和 `VALID_INTENTS` 同步扩展。

- **F2**: 分析意图智能检测——用户无需说"分析"关键词也能触发。自然提问如"Shokz 最近表现怎么样"、"美妆品类什么趋势"、"Tier 2 整体健康度如何"应被 LLM 识别为 `analysis` 意图，并提取分析对象类型（merchant/category/tier）和目标实体名称。

- **F3**: 商户维度分析——输入目标商户名称，输出：(a) 商户核心指标概览表（EPC、AOV、CVR、订单量、佣金等，含排名百分位），(b) 与同品类/同 Tier 商户的横向对比，(c) 强弱项摘要（亮点、短板、支付风险），(d) LLM 生成的行动建议。

- **F4**: 品类维度分析——输入目标品类名称（如"美妆"/"Electronics"），输出：(a) 品类整体统计（商户数、总收入、平均转化率、平均 EPC），(b) 品类内 Top/Bottom 商户排名，(c) 与全站均值的对比，(d) LLM 生成的品类健康度评估。

- **F5**: Tier 维度分析——输入目标 Tier（如"Tier 2"），输出：(a) Tier 内核心指标聚合统计，(b) 与其他 Tier 的横向对比（指标差异），(c) Tier 内商户分化情况（头部/腰部/尾部），(d) LLM 生成的升降级建议。

- **F6**: LLM 生成分析文字——将结构化分析数据（JSON 格式的统计摘要）发给 LLM API，让 LLM 生成自然语言分析叙述和建议。复用 `llm_classify.py` 的 provider 配置（DeepSeek/Claude），新建 `/api/chat/analyze` 端点。前端收到 LLM 文字后渲染到分析回答中。

- **F7**: 分析结果呈现——分析回答采用「统计表格 + 文字解读」混合布局：上方为结构化数据表格（复用现有表格渲染风格），下方为 LLM 生成的分析叙述。表格始终显示（即时渲染），LLM 文字可在加载完成后追加。

- **F8**: LLM 不可用时的降级——当 LLM API 不可用（超时/错误/未配置 API key）时，分析表格正常显示，文字解读部分降级为模板生成的简要结论（基于数据特征判断，如"高于均值"、"排名靠前"等固定表达）。

## 非功能需求

- **N1**: 延迟——分析表格（纯前端计算）应在用户发送消息后 500ms 内渲染。LLM 分析文字的 P95 延迟（含网络往返）应 ≤ 10 秒，超时上限 15 秒。

- **N2**: 可靠性——LLM 分析服务的可用性不影响分析表格的展示。任何 LLM 故障都应透明降级到模板文字。模板路径不依赖任何外部 API。

- **N3**: 安全性——`/api/chat/analyze` 端点需要 session 认证。请求体仅包含结构化的统计摘要数据（不含任何用户身份信息或原始商户数据），大小 ≤ 8KB。API key 仅存储在服务端。

- **N4**: 成本控制——LLM 分析请求发送的是结构化的统计摘要（JSON 格式的数字和标签），不发送原始 offer 数据。每次分析请求的 prompt 控制在 1500 token 以内。使用和意图分类相同的 provider 和模型。

- **N5**: 兼容性——复用现有 `llm_classify.py` 的 provider 配置和 API 调用逻辑。前端改动遵循现有 vanilla JS 风格，不引入新框架或图表库。遵循项目的 dual runtime 架构（`server.py` + `api/` Vercel handler）。

- **N6**: 国际化——分析模板文字和 LLM 生成的回答支持中英双语。前端根据当前语言设置（`state.language`）发送语言参数给 LLM，确保分析文字与界面语言一致。

## 不做的事

- 不做趋势/时间序列分析（需要历史数据，需后端 DB 支持，后续迭代考虑）
- 不做数据可视化图表（柱状图、雷达图等，不引入图表库）
- 不做支付维度的独立分析（支付分析逻辑已在现有 payment intent 中，本期不重复）
- 不改变现有意图的回答逻辑——asin/merchant/payment/recommendation/tier/category 的回答保持不变
- 不做多轮对话中的上下文分析追问（如"那这个商户呢？"）
- 不做批量分析（一次分析多个商户/品类）

## 验收标准

- **AC1**（对应 F1）：LLM 能正确分类分析类意图——输入"分析 Shokz"、"美妆品类趋势如何"、"Tier 2 整体表现"分别返回 `intent: "analysis"` 及对应的分析对象类型和实体名

- **AC2**（对应 F2）：自然语言隐式触发——输入"Shokz 最近怎么样"（不包含"分析"二字）返回 `analysis` 意图

- **AC3**（对应 F3）：商户分析完整性——分析 Shokz 时，回答包含核心指标表、同品类对比数据、强弱项摘要

- **AC4**（对应 F4）：品类分析完整性——分析 Electronics 时，回答包含品类统计、品类内排名、与全站均值对比

- **AC5**（对应 F5）：Tier 分析完整性——分析 Tier 2 时，回答包含聚合统计、跨 Tier 对比、内部分化情况

- **AC6**（对应 F6）：LLM 生成分析文字——分析请求返回后，聊天中显示由 LLM 生成的自然语言叙述和行动建议

- **AC7**（对应 F7）：混合呈现布局——分析回答中统计表格在 LLM 文字之前渲染，表格始终显示

- **AC8**（对应 F8）：LLM 降级——LLM API 不可用时，分析表格正常显示，文字部分降级为模板生成的简要结论（如"该商户 AOV 高于品类均值，转化率处于中上水平"）

- **AC9**（对应 N1, N2）：现有测试全量通过——`test_chatbot_intent_flow.mjs` 和 `test_zh_chatbot.mjs` 全部通过，现有意图分类不受影响
